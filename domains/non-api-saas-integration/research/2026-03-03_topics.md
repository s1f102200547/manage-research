## 拡張ブレインスト―ミング：APIなしSaaS統合の全可能性

### 🧠 知覚層（データをどう「読む」か）

**A. DOM/HTML解析**
- HTMLをそのままLLMに食わせて構造理解
- Shadow DOMやiframeも含めた完全なページ表現の標準化
- 「意味的HTML」としてLLMが解釈するフレームワーク

**B. アクセシビリティツリー活用**
- ARIA属性、role、labelをセマンティックに解析
- スクリーンリーダー用のツリーはUIより安定している
- Playwrightの`page.accessibility.snapshot()`がすでにこれ

**C. Vision + Layout理解**
- PixelレベルでなくGestallt的なUI理解（グルーピング、近接、整列）
- GPT-4V / Claude Vision でUI意図を推論
- 「このボタンは何をする？」を画像から判断

**D. ネットワークトラフィック解析**
- XHR/Fetchの傍受でSaaSの「隠れAPIエンドポイント」を発見
- GraphQLリクエストのリバースエンジニアリング
- WebSocketストリームの解読

**E. メモリ/プロセス解析**
- ブラウザのメモリダンプから構造化データを抽出（研究レベル）
- Electronアプリなら特に有効

---

### 🖐 操作層（どう「書く」か）

**F. 意図ベース操作（Intent-driven UI）**
- 「在庫を3に変更したい」という意図をLLMが解釈してDOM操作に変換
- セレクタでなく**意味**で操作する
- UIが変わっても意図が合えば動く

**G. 操作の記録・再生（Operation Recording）**
- 人間の操作を一度記録して「テンプレート化」
- 変数部分だけLLMが埋める
- Playwright codegen的だが、LLMで汎化する

**H. 差分検出による自己修復**
- UIが変わったことを検知して自動的に操作を再学習
- 「前回は動いたのに今回動かない」を自律修復するエージェント

**I. フォームの意味理解**
- フォームフィールドの意味をLLMが推論して自動入力
- ラベル・プレースホルダー・バリデーションエラーから意図を逆算

---

### 🔌 統合層（どう「繋ぐ」か）

**J. ローカルプロキシとしてのブラウザ拡張**
- Chrome拡張がSaaSとエージェントの橋渡しをする
- ユーザーのセッションを使うので認証問題が消える
- SaaSのUIに「APIレイヤー」を後付けする発想

**K. サービスワーカー介入**
- PWAのService Workerレベルでリクエストを傍受・変換
- ブラウザ内でMITMを合法的に実現

**L. 標準化レイヤー（Universal Adapter）**
- 各SaaSのUIパターンを「抽象操作セット」に正規化
- CRUD操作に変換するアダプターをSaaSごとに定義
- OSSコミュニティでアダプターを共有するエコシステム

**M. LLMによるSaaS仕様の自動生成**
- SaaSのUIを探索してOpenAPI仕様を自動生成
- 「仮想API」をリバースエンジニアリングで作る

---

### 🌐 環境・インフラ層

**N. ヘッドレスブラウザのクラウド化**
- Browserbaseのような「ブラウザをAPIとして提供するインフラ」
- スケーラブルなBrowser Useの基盤

**O. デジタルツイン的なSaaSミラーリング**
- SaaSの状態をローカルDBに常時同期してキャッシュ
- 読み取りはキャッシュから、書き込みは非同期でSaaSに反映

**P. マルチエージェント協調**
- 各SaaSに専門エージェントを配置
- オーケストレーターが意図を各エージェントに委譲
- 旅館文脈：「じゃらんエージェント」「一休エージェント」が協調

### 🔬 研究レベルの可能性

**Q. UI仕様の業界標準化（政策レベル）**
- 「APIなしSaaSはデジタル庁認定を受けられない」みたいな規制誘導
- 実際にEUはPSD2でFinTechのAPI開放を義務化した前例がある

**R. SaaSベンダーへのインセンティブ設計**
- APIを開放したベンダーには「AIエージェント対応認定」を付与
- プラットフォーム側がAPI開放を促すビジネスモデル

**S. WASM/サンドボックス内でのSaaS実行**
- SaaSのフロントエンドをサンドボックスで実行してAPIとして包む
- 理論上は可能、実用化は遠い

## メタ視点：問題の本質は何か

全アプローチを俯瞰すると、実は3つの異なる問題が混在してる。

```
1. 認識問題　→「SaaSの状態をどう知るか」
2. 操作問題　→「SaaSをどう動かすか」
3. 標準化問題 →「異なるSaaSをどう統一的に扱うか」
```

**3が一番難しくて一番価値がある。**

Zapierが解いたのも3だった。ただZapierはAPIがある前提だった。**APIなしで3を解く**のがお前のいる場所。
