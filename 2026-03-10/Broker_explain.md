少し抽象度が高いので、**3つの仕組み（Adapter / Queue / Routing）だけに分解して理解すると一気に分かる**。
EAIのBrokerの中で実際にやっている仕事は **この3つだけ**。🧠

---

# 1. Adapter（翻訳機）

問題：
システムごとに **データの形が違う**。

例

**OTA**

```json
{
 "check_in_date": "2025-03-10",
 "guest_count": 2
}
```

**旅館システム**

```json
{
 "入館日": "250310",
 "人数": "2名"
}
```

このままだと互いに理解できない。

そこで **Adapterが翻訳する。**

```
OTA形式
↓
Adapter
↓
共通フォーマット
↓
Adapter
↓
旅館形式
```

つまり

**Adapter = データ翻訳機**

---

## Canonical Data Model（共通言語）

ここが重要。

もし10個のシステムがあるとする。

### 直接翻訳

```
A↔B
A↔C
A↔D
B↔C
B↔D
C↔D
...
```

必要な翻訳

[
N(N-1)/2
]

例

| システム数 | 翻訳数 |
| ----- | --- |
| 5     | 10  |
| 10    | 45  |

爆発する。💥

---

### 共通言語を使う

```
A → 共通フォーマット
B → 共通フォーマット
C → 共通フォーマット
```

必要な翻訳

[
N
]

例

| システム数 | 翻訳数 |
| ----- | --- |
| 5     | 5   |
| 10    | 10  |

これが

**Canonical Data Model（CDM）**

つまり

> 全システムが「共通言語」を話す

---

# 2. MOM（Message Queue）

次の問題。

もしシステムAが送信したとき
システムBが止まっていたら？

普通の通信

```
A → B
```

Bがダウン → **データ消える**

---

そこで

**キュー（Queue）**を使う。

```
A → Queue → B
```

### 具体例

```
予約システム
↓
「予約入りました」
↓
Queue
↓
会計システム（あとで読む）
```

重要な性質

* 相手が止まっていてもOK
* 後で処理できる
* メッセージが保存される

この方式を

**非同期通信（Asynchronous）**

という。

例：

| 通信  | 例   |
| --- | --- |
| 同期  | 電話  |
| 非同期 | メール |

---

# 3. Routing（仕分け）

次の問題。

1つのイベントを
**複数システムに送る必要がある。**

例

予約イベント

```
予約
↓
CRM更新
↓
会計
↓
シフト管理
```

そこで

**Router**

が判断する。

```
予約イベント
↓
Router
├ CRM
├ 会計
└ シフト
```

この判断方法はいくつかある。

### Content-Based Router

内容で判断

例

```
if country = US
   → USシステム
else
   → 日本システム
```

# 4. Brokerの中身

今の話をまとめると

Brokerはこれ。

```
          ┌───────────┐
System A ─┤ Adapter   │
System B ─┤ Adapter   │
System C ─┤ Adapter   │
          │           │
          │  Queue    │
          │           │
          │  Router   │
          └───────────┘
```

つまり

Broker =

1️⃣ Adapter（翻訳）
2️⃣ Queue（保存）
3️⃣ Router（配送）
