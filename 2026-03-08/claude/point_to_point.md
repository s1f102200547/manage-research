## Point-to-Point接続の時代：論文・書籍ベースの調査と初心者向け解説

---

### 1. 学術的背景：主要な文献

この時代の問題を最も体系的に論じたのは、以下の文献群です。

**David S. Linthicum『Enterprise Application Integration』（1999年, Addison-Wesley）** — EAIという概念を体系化した最初期の書籍で、バラバラなレガシーシステム、ハードウェア・OS・ネットワーク技術の寄せ集め、独自仕様のパッケージアプリケーションなどの統合課題を包括的に扱った。Linthicumは「EAI」という技術用語の生みの親とも言われています。

**Gregor Hohpe & Bobby Woolf『Enterprise Integration Patterns』（2003年）** — EIPの前提として、「なぜ企業がこんな状態に陥るのか」に触れ、ビジネスアプリケーションの構築は難しく、単一の巨大アプリケーションで企業全体を動かすのはほぼ不可能であり、SAP・Oracle・PeopleSoftのようなERP大手ですら典型的な企業が必要とする業務機能の一部しかカバーできないと述べました。

**Ruh, Maginnis & Brown『Enterprise Application Integration: A Wiley Tech Brief』（2001年）** および **Johnson『Enterprise Software System Integration: An Architectural Perspective』（2002年）** も、Point-to-Point統合の限界とミドルウェアの必要性を論じた重要文献です。

---

### 2. Point-to-Pointとは何か？（初心者向け）

**たとえ話：電話線の時代**

1990年代の企業システム統合を、「電話をかけるのに、相手ごとに専用の電話線を引く」状態だと想像してください。

- AさんとBさんの間に1本の線
- AさんとCさんの間にもう1本
- BさんとCさんの間にもう1本

3人なら3本で済みます。でも10人になると45本、20人なら190本必要です。これがPoint-to-Point接続です。

---

### 3. なぜPoint-to-Pointが生まれたのか：「ストーブパイプ」の歴史

長年にわたり、情報システムは「ストーブパイプ」（煙突型）として構築されてきた。それぞれが独立した自己完結型アプリケーションで、他のアプリケーションとの統合を想定して設計されていなかった。

つまり、1980年代〜90年代初頭の企業では：

- 会計システムは会計部門のためだけに作られた
- 在庫管理システムは倉庫部門のためだけに作られた
- 受注システムは営業部門のためだけに作られた

それぞれ異なるベンダー、異なるOS、異なるデータベース、異なるデータ形式で動いていました。企業のIT環境に対する依存が複雑化・拡大するにつれ、バラバラなアプリケーションを統一的な業務プロセスに統合する手法の必要性が優先課題として浮上した。世代を重ねた技術で自動化の島を作った後、ユーザーやビジネスマネージャーはそれらを繋ぐシームレスな橋を求め始めた。

そこで最初に取られたアプローチが、2つのシステムを直接つなぐPoint-to-Point方式でした。

---

### 4. Point-to-Pointの具体的な仕組み

Point-to-Point統合アーキテクチャでは、2つのアプリケーションが通信する必要がある場合、接続が確立され、コネクタがアプリケーションAとBの間の通信を可能にする。

当時の技術的手段は主に4つありました（Linthicumの4レベル分類）：

**① データレベル統合（Data-Level EAI）**
- 直接データベースにアクセスしてデータを抜き出す方式
- 例：在庫DBからODBC/JDBC経由でSQLで読み取り、会計DBに書き込む
- 最もシンプルだが、相手のDBスキーマ（テーブル構造）を知っている必要がある

**② アプリケーションインターフェースレベル（Application Interface-Level EAI）**
- アプリが公開するAPI（Application Programming Interface = アプリ同士が情報をやりとりするための取り決め）を使う
- 例：SAPのBAPI、PeopleSoftのAPIなど
- ただし当時のパッケージソフトはAPIが貧弱で独自仕様が多かった

**③ メソッドレベル（Method-Level EAI）**
- 業務ロジック（メソッド）そのものを共有する
- 分散オブジェクト技術（CORBA = Common Object Request Broker Architecture、COM = Component Object Modelなど）を使用
- 技術的に高度で、実装コストが高い

**④ ユーザーインターフェースレベル（User Interface-Level EAI）**
- ユーザーインターフェースレベルEAIでは、ユーザーインターフェースそのものがEAIのインターフェースとなる。「スクリーンスクレイピング」として知られるプロセスで、画面上のデータを読み取って別のシステムに入力する
- メインフレームの端末画面をHLLAPI（High Level Language API）で操作する手法が中心
- **これがまさに現代のRPAやAIエージェント統合の原型**

### 5. Point-to-Pointの何が問題だったのか

#### 問題①：接続数の爆発（N×(N-1)/2問題）

システムの数が増えると接続数はn(n-1)/2の公式で増加する。5システムで10接続、10システムで45接続、20システムでは190接続になる。この指数的増加が「スパゲティアーキテクチャ」につながる。

#### 問題②：変更の連鎖的影響

P2P統合ではシステム同士がAPI、データ構造、プロトコルに直接依存する。一方のシステムがAPIを更新したりフィールド名を変更すると、接続されたすべての統合が壊れる可能性がある。この密結合が全体を脆弱にする。

#### 問題③：コスト

従来のミドルウェアの経済性は大半のIT組織にとってEAIを手の届かないものにしていた。Aberdeen Groupによれば、単純な2アプリケーション間の連携でさえ1,000万ドルもの費用がかかる場合があった。

#### 問題④：レガシーの固定化

スパゲティ統合されたアプリケーションは置き換えがほぼ不可能になる。変更の影響範囲の見積りがしばしば桁違いに間違う。さらに、主要システムを置き換える場合、新システムが古い接続をすべて引き継ぐのではなく、旧システムがすべての既存スパゲティ接続のハブとして残り続け、新システムは旧システムからのデータフィードやETLで動く、という事態が起きる。つまり旧システムは永遠に残る。

#### 問題⑤：ミドルウェア自体のPoint-to-Point性

メッセージキューイングやRPCを使う従来のミドルウェアはPoint-to-Pointのソリューションしか提供しない。つまりシステムAとBの間のリンクでしかない。追加のシステムを接続しようとすると、すぐにミドルウェアリンクの複雑な絡まりになる。さらに悪いことに、従来のミドルウェアはソースとターゲットの両システムに大幅な変更を要求する。

---

### 6. 実際にはどんな「スパゲティ」が起きていたか

スパゲティは複数の形態をとるが、しばしばすべての形態が含まれ、複数世代の技術が混在する。過剰なPoint-to-Point接続、密結合の接続技術、データベーストリガー、EAIプロセスステップに埋め込まれたビジネスロジック、多数のバッチ処理、ETLロードと抽出、ほぼ同一（だが完全には同一でない）データセットを提供する複数のサービス、多数のメッセージキューなどが含まれる。

---

### 7. Point-to-Pointから次の時代への転換点

この限界を受けて生まれたのが「中央にハブを置く」というEAIの発想でした。

EAIは従来のミドルウェアのアプローチとは根本的に異なる統合手法を提供する。EAIはビジネスプロセスとデータの両方が、アプリケーションを横断して相互に会話できる共通の方法を作り出す。

また、Hohpe & Woolfの観点では、4つの統合スタイル（File Transfer, Shared Database, RPC, Messaging）のうち、Messaging（非同期メッセージング）こそがPoint-to-Pointの限界を根本的に解決する方式だと主張しました。

---

### 8. Haruの研究文脈への接続

Linthicumが1999年に定義した4つのEAIレベルのうち、**第4レベル「User Interface-Level EAI」（スクリーンスクレイピング）** は、まさにHaruの卒研テーマの直接的な先祖です。

当時の「スクリーンスクレイピング」は、メインフレームの3270端末エミュレータからHLLAPIでテキストを読むという非常に限定的な技術でした。それが、現代では：

- **RPA**（UiPath等）：GUIのボタン・フィールドをルールベースで操作
- **AI Agent**（Browser Use + LLM）：DOMやアクセシビリティツリーをLLMが解釈し、UIが変わっても自律的に適応

と進化しています。つまりHaruの三層フレームワーク（認識・操作・標準化）は、1999年にLinthicumが「最も原始的だが唯一の手段」として記述したUI-Level EAIを、25年分の技術進化（LLM + ブラウザ自動化）で再発明する試みと位置づけられます。
